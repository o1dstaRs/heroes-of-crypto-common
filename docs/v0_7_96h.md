# v0.7 96-hour autonomous improvement run

This run is a research-only evolutionary policy-search program around the existing wide-action RAWS search.
It may produce a candidate artifact and statistical evidence. It must not edit Git-tracked source, commit,
push, bake weights, change the v0.7 default, or deploy anything.

The Linux supervisor is `scripts/run_v0_7_96h.sh`. It is intentionally separate from the optimizer entry
point, `src/simulation/optimizer/v0_7_96h.mjs`: the supervisor owns process lifetime, while the optimizer owns
checkpointing, phase transitions, compute throttling, and any coexistence with a running League job.

## Exact target panel

The requested four cohorts are the committed `V07_ARCHETYPES`, not the acceptance harness's separate
`melee/mixed/random` distribution cohorts. Each fixed template gives both seats the identical six-stack,
experience-budget roster under the LiveTwin SEE_NONE setup. The level slots are `L1,L1,L2,L2,L3,L4`.

| Cohort     | Template              | Exact roster                                                   |
| ---------- | --------------------- | -------------------------------------------------------------- |
| Mage       | `mage_frontline`      | Squire, Berserker, Healer, Satyr, Crusader, Hydra              |
| Mage       | `mage_fireline`       | Arbalester, Centaur, Healer, Satyr, Cyclops, Tsar Cannon       |
| Melee Mage | `melee_magic_utility` | Peasant, Wolf Rider, Harpy, Valkyrie, Ogre Mage, Angel         |
| Melee Mage | `melee_magic_brawler` | Squire, Berserker, Troll, Harpy, Ogre Mage, Behemoth           |
| Aura       | `aura_support`        | Peasant, Leprechaun, White Tiger, Valkyrie, Griffin, Pegasus   |
| Aura       | `aura_offense`        | Wolf Rider, Leprechaun, White Tiger, Valkyrie, Crusader, Angel |
| Ranged     | `ranged_precision`    | Arbalester, Centaur, Elf, Medusa, Cyclops, Tsar Cannon         |
| Ranged     | `ranged_control`      | Orc, Arbalester, Beholder, Elf, Cyclops, Gargantuan            |

The primary point estimate is the equal-template decisive win rate against the frozen v0.6 opponent, with
paired side swaps and Normal-map final cells. Optimization ranks the minimum of all eight template scores;
it must not hide a weak sibling inside a strong cohort average.

- **Stretch 90:** every one of the four equal-template cohort point estimates is at least 90%.
- **Certified 90:** every one of the four equal-template cohort aggregates has a simultaneous one-sided
  95% paired-cluster lower bound of at least 90%, using the conservative eight-claim Bonferroni critical value.
- **Strict all-aspects 90:** every one of the eight template point estimates is at least 90%.

Reports must keep these labels distinct. A point estimate above 90% is not a certified lower bound. Draws are
also scored as one half for an anti-gaming diagnostic, and draw-or-Armageddon must stay at or below 1%.

## Program phases

1. **Probe:** reproduce the incumbent and measure the initial exact eight-template RAWS frontier. Race
   preregistered leaf, gate, horizon, rollout-count, and bounded-candidate settings. A selected probe
   champion whose template-level upper diagnostic remains below 90% records a probe-only gap; it does not
   claim that later CEM candidates or the overall search space cannot reach the target.
2. **Evolutionary leaf tuning:** use CEM around the strongest model leaf while jointly tuning the override
   gate, horizon, rollout count, and bounded candidate caps. Successive-halving races share scenarios within
   a stage and rotate seed banks between stages and generations. No source-generating or self-modifying agent
   is part of this phase.
3. **Freeze:** stop optimization with at least 36 hours left. Freeze one candidate fingerprint containing
   source revision, weights, search settings, environment, training manifests, and artifact hashes. Final
   seeds cannot be opened before this fingerprint exists.
4. **Final:** evaluate the frozen candidate once on untouched 12,000-game-per-template v0.6 seeds, then run
   a separate 2,000-game-per-template v0.4 transitivity panel. Both final panels always receive the declared
   `V07_96H_WORKERS=20` budget even if League still exists. The evaluator runs at most four templates at a
   time, assigning five workers to each template in a full 20-worker batch. Reading either final result never
   reopens tuning. Generated-roster, extra-map, and committed-default acceptance remain required follow-up
   before bake.

Search-enabled output must be called `v0.7+RAWS` until it is deliberately baked and the clean-default v0.7
acceptance protocol is rerun. The 96-hour job itself cannot authorize that promotion.

## Seed and overfit controls

Before the first outcome, the optimizer persists one immutable manifest containing separate probe, rotating
generation, and final panels. Seeds are derived from a run-specific commitment and the panel/template tuple;
colliding streams are deterministically rehashed. Every direct and side-swap-derived seed is checked against
the other 96-hour panels, every committed v0.7 evidence manifest, and sibling 96-hour manifests under the
same run parent directory. The committed `v0_7_prior_zinc_seed_denylist.json` also expands and excludes
144,600 unique pair seeds opened or conservatively reserved by the recoverable mega-validation, acceptance,
fixed-archetype, guard, CEM, caster, and RAWS streams that materially informed this protocol.
The concurrently running draft-genome League is not an input to this fixed-roster RAWS search and remains
outside this freshness boundary; its process is considered only for research-phase CPU throttling.
Allocation of a sibling manifest is one locked transaction under that parent:
the production `.v0_7_96h_seed_allocation.flock` kernel lock covers the sibling scan, collision resolution,
immutable manifest write, and fsync in a short initialization subprocess. Process death releases the lock;
no stale pathname is automatically reclaimed. Dirty local smoke uses a separate fail-closed lock file.

- Candidate races share scenarios within one race, but training banks rotate between generations.
- Each scout, mid, deep, and final bank is consumed for its declared purpose only. The final panels are not
  evaluated until the candidate fingerprint has been frozen immutably.
- Revision-, environment-, run-, panel-, seed-, and behavior-bound checkpoints preserve completed work
  across process restarts. Large cells are divided only at even game boundaries, so every resumable subshard
  contains complete side-swap pairs; the evaluator defaults to 200 games, or 100 pairs, per subshard. Each
  subshard writes its own audit fragment; resume validates shard identity and hashes before deterministic
  report and audit assembly, without rerunning completed pairs.
- Fixed-template final cells preserve equal rosters, complete side-swapped pairs, and side balance.
- Final confidence applies the conservative eight-claim Bonferroni critical value to each of the four
  equal-template archetype aggregates, matching the implemented target assessment.

If the wall-clock evaluation window closes before the primary final is complete, the optimizer still emits
the structured, self-hashed `TERMINAL.json`. It records `finalStatus: "primary_incomplete_deadline"`, marks
the incomplete panel as unqualified, and sets `targetGate.achieved` and every 90% target flag to `false`.
Partial subshards remain resumable evidence, but they never become a target claim.

## Candidate gates

A candidate may replace the research incumbent only when a fresh selection bank shows all of the following:

- the minimum-template score improves by at least 0.5 percentage points;
- no template regresses by more than 1 percentage point;
- candidate engine rejections and missing rejection counts are both zero;
- draw-or-Armageddon is at most 1%;
- p95 and maximum per-match search time fit inside the server's 240-second per-match budget;
- the candidate, source, configuration, manifest, and result fingerprints are complete and stable.

Failure of any final gate leaves the current v0.7 default untouched. The report may still retain the best
validated research candidate and its exact gap to target.

## Expected configuration

The wrapper requires one unique run directory, either as its first argument or as `V07_96H_OUT`. Relative
paths are resolved from the repository root. It passes both `V07_96H_OUT` and `--out=<absolute-path>` to the
optimizer. The expected run configuration is:

```bash
export V07_96H_OUT="$HOME/hoc-common-v07-96h-runs/run-$(date -u +%Y%m%dT%H%M%SZ)"
export V07_96H_HOURS=96
export V07_96H_WORKERS=20
export V07_96H_COEXIST_WORKERS=4
export V07_96H_FINAL_RESERVE_HOURS=36
export V07_96H_CHECKPOINT_GAMES=200
export V07_96H_POP=12
export V07_96H_ELITE=3
export V07_96H_MAX_GENERATIONS=64
```

The optimizer detects the current League process without inspecting, renicing, pausing, or killing it.
Probe and evolutionary phases use `V07_96H_COEXIST_WORKERS` while League is active, then move toward
`V07_96H_WORKERS` when capacity is available. Final v0.6 and v0.4 panels never use the coexistence throttle:
they always use the declared 20-worker budget. On the current 24-core/48-thread Zinc host, 4 coexistence
workers during research and 20 workers after League or during final panels are the protocol defaults.
`V07_96H_CHECKPOINT_GAMES` must be an even integer of at least 2. It is persisted in `run.json`, passed to
every evaluator as `--checkpoint-games`, and checked against the resulting report; a resume or report with a
different subshard size fails closed.

Supervisor controls have safe defaults:

| Variable                       | Default                                 | Meaning                                |
| ------------------------------ | --------------------------------------- | -------------------------------------- |
| `V07_96H_OPTIMIZER`            | `src/simulation/optimizer/v0_7_96h.mjs` | Alternate optimizer entry point        |
| `V07_96H_DEADLINE_EPOCH`       | start plus `V07_96H_HOURS`              | Explicit UTC epoch deadline            |
| `V07_96H_TERMINAL_MARKER`      | `TERMINAL.json` under the run directory | Optimizer-owned atomic terminal marker |
| `V07_96H_HEARTBEAT_SECONDS`    | `30`                                    | Supervisor poll and heartbeat interval |
| `V07_96H_RESTART_BASE_SECONDS` | `15`                                    | First nonzero-exit restart delay       |
| `V07_96H_RESTART_MAX_SECONDS`  | `900`                                   | Exponential-backoff ceiling            |
| `V07_96H_MAX_RESTARTS`         | `8`                                     | Consecutive-failure stop limit         |
| `V07_96H_STOP_GRACE_SECONDS`   | `30`                                    | TERM grace before KILL                 |

The wrapper always exports `V07_96H_RESEARCH_ONLY=1` and launches the optimizer through `nice -n 10`. The
optimizer must fail closed if research-only mode is absent. The supervisor does not trust terminal-file
existence alone: it validates schema version 1, `status: "complete_research_only"`, equality with the immutable
`run.json` run ID, and the terminal's SHA-256 over canonical content excluding `terminalSha256`. An invalid or
stale marker is a protocol error. A zero exit without a valid atomic terminal marker is also a protocol error;
nonzero exits restart with bounded exponential backoff until a valid marker, persisted deadline, or
eight-failure stop limit appears.

Because this repository intentionally has no tracked Bun lockfile, `run.json` records the complete installed
dependency layout: relative package location, name, version, package.json hash, and a canonical aggregate
hash. Every resume recomputes that snapshot and fails on drift. This makes the exact resolved closure used by
the run auditable without adding the developer's protected local `bun.lock` to Git.

## Start and operate on Zinc

From a newly cloned, clean Zinc checkout whose `main` equals `origin/main`:

```bash
cd "$HOME/hoc-common-v07-96h"
bun install --no-save
test -z "$(git ls-files -- bun.lock bun.lockb)" && rm -f bun.lock bun.lockb
test -z "$(git status --porcelain --untracked-files=all)"
OUT="$HOME/hoc-common-v07-96h-runs/run-$(date -u +%Y%m%dT%H%M%SZ)"
nohup env \
  V07_96H_OUT="$OUT" \
  V07_96H_HOURS=96 \
  V07_96H_WORKERS=20 \
  V07_96H_COEXIST_WORKERS=4 \
  V07_96H_FINAL_RESERVE_HOURS=36 \
  V07_96H_CHECKPOINT_GAMES=200 \
  scripts/run_v0_7_96h.sh >/dev/null 2>&1 &
printf 'launcher pid=%s out=%s\n' "$!" "$OUT"
```

Status and logs:

```bash
cat "$OUT/supervisor.heartbeat"
test -f "$OUT/heartbeat" && cat "$OUT/heartbeat"
cat "$OUT/supervisor.pid" "$OUT/optimizer.pid"
ps -fp "$(cat "$OUT/supervisor.pid")" "$(cat "$OUT/optimizer.pid")"
tail -n 100 "$OUT/supervisor.log"
tail -n 100 "$OUT/optimizer.log"
tail -n 100 "$OUT/driver.log"
test -f "$OUT/TERMINAL.json" && cat "$OUT/TERMINAL.json"
```

Stop the supervisor, not only its child (the supervisor intentionally restarts a killed child):

```bash
kill -TERM "$(cat "$OUT/supervisor.pid")"
```

Restart the same run without extending its persisted deadline:

```bash
nohup env V07_96H_OUT="$OUT" scripts/run_v0_7_96h.sh >/dev/null 2>&1 &
```

Supply the same optimizer configuration on manual restart. The optimizer compares it with its frozen run
metadata and fails closed on drift. Completed even-pair subshards and their audit fragments resume in place.
Reusing the path while a supervisor is live exits with status 75 due to `flock`. Reusing it after a valid
`TERMINAL.json` or `SUPERVISOR_DEADLINE` exists is an idempotent no-op; an invalid terminal marker fails
closed instead of suppressing the optimizer.

Persistent supervisor files are `supervisor.log`, `optimizer.log`, `supervisor.pid`, `optimizer.pid`,
`supervisor.heartbeat`, `supervisor.started_epoch`, `supervisor.deadline_epoch`, and `supervisor.lock`.
`SUPERVISOR_DEADLINE` records a wall-clock stop without impersonating the optimizer's terminal verdict;
`SUPERVISOR_PROTOCOL_ERROR` records a clean child exit that omitted its required verdict marker.
The optimizer's own atomic status is `heartbeat`; its structured progress log is `driver.log`. A complete
or deadline-incomplete optimizer conclusion is the validated, run-bound, self-hashed `TERMINAL.json`.
